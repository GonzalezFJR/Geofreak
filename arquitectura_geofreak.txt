ARQUITECTURA BASE PROPUESTA — WEB DE JUEGOS DE GEOGRAFÍA

1. OBJETIVO
Arquitectura inicial para una web de juegos de geografía con:
- juego individual
- perfiles de usuario
- estadísticas
- tests tipo quiz
- rankings
- duelos y torneos
- base preparada para evolución posterior
- soporte en dos idiomas: inglés y español

2. STACK PRINCIPAL
- Frontend web
- Backend Python con FastAPI
- Despliegue inicial en AWS EC2
- Una única instancia de backend
- DynamoDB como base de datos principal operativa
- S3 para almacenamiento de objetos, assets y eventos analíticos
- WebSockets para duelos
- Sin Redis / ElastiCache en esta fase (sí en la fase 2, pero por ahora nada de esto)

3. COMPONENTES
3.1 Frontend
- cliente web
- consume API REST del backend
- consume WebSockets para duelos / torneos
- usa cookies técnicas para sesión/autenticación
- usa almacenamiento local sólo para preferencias no sensibles y caché ligera de cliente
- inglés y español

3.2 Backend FastAPI
Responsabilidades:
- autenticación y autorización
- gestión de usuarios y perfiles
- gestión de amigos, invitaciones, partidas y torneos
- lógica de juegos
- cálculo de puntuaciones
- actualización de estadísticas agregadas
- exposición de API REST
- WebSockets para duelos y torneos
- generación y consulta de leaderboards
- registro de eventos analíticos en S3

3.3 AWS EC2
- ejecución del backend FastAPI
- uso de Nginx como reverse proxy
- despliegue inicial monoinstancia
- sin dependencia de almacenamiento persistente local para datos críticos

6. COOKIES Y ALMACENAMIENTO LOCAL
6.1 Cookies
Usar sólo para:
- sesión/autenticación
- refresh/session token si aplica
- preferencia de consentimiento
- preferencias técnicas estrictamente necesarias

No usar cookies para:
- publicidad
- perfilado publicitario
- tracking comercial innecesario

6.2 localStorage / sessionStorage
Usar sólo para:
- tema visual
- idioma
- preferencias de interfaz
- estado UI no sensible
- caché ligera de cliente

No guardar en localStorage:
- passwords
- tokens sensibles si se evita
- datos críticos de autorización
- datos privados de usuario que deban estar sólo en servidor

8. ESTADO DE PARTIDA Y CADUCIDAD
8.1 Duelos
- persistencia en DynamoDB
- actualización en tiempo real vía WebSockets sobre una única instancia
- caducidad de estado: 8 horas
- guardar expires_at para expiración lógica
- guardar estadísticas asociadas en DynamoDB y/o S3

8.2 Torneos
- persistencia en DynamoDB
- conjunto de N pruebas para 2 o más jugadores
- puntuaciones acumuladas persistentes
- caducidad de estado: 24 horas
- guardar expires_at para expiración lógica
- guardar estadísticas asociadas en DynamoDB y/o S3

9. PARTIDAS ONLINE Y TORNEOS
9.1 Solo
- backend genera preguntas/config
- backend corrige respuestas
- backend guarda resultado final
- backend actualiza user_stats

9.2 Duelo / multijugador simple
- modo: “same questions, same match, no strict timer”
- dos o más jugadores comparten el mismo match_id
- backend entrega el mismo set de preguntas
- cada jugador responde a su ritmo
- backend guarda progreso y puntuación parcial en DynamoDB
- clientes reciben actualizaciones vía WebSockets
- backend calcula resultado final y actualiza estadísticas

9.3 Torneos
- conjunto de N pruebas para 2 o más jugadores
- persistencia base en DynamoDB:
  - estructura del torneo
  - jugadores
  - rondas
  - puntuaciones acumuladas
  - estado general
- backend actualiza clasificación tras cada prueba
- backend guarda resultados finales y estadísticas derivadas

17. JUEGOS PRINCIPALES (para empezar a crear)
- Pruebas de tipo "solo", "duelo" y "torneo": dos pruebas principales. 1. Orden según característica. Se muestran 5 entidades al azar y se escoge una característica (población, superficie, PIB, etc.) y el jugador debe ordenarlas de mayor a menor o viceversa. 2. Prueba de comparación (entidad 1) vs (entidad 2) en alguna característica al azar. Ejemplo "¿Qué país tiene más población?" y se muestran dos cards. La prueba consiste en 10 preguntas de este estilo. Se reporta una puntuación final. Estas dos pruebas se pueden jugar en modo "solo", en modo "duelo" (cada jugador responde en su PC o móvil, pregunta a pregunta, y se pasa a la siguiente pregunta cuando todos responden, mostrando respuestas correctas, quién acertó y quién falló...). Y en modo "torneo" (conjunto de N pruebas, por ejemplo 5 pruebas, que pueden ser de tipo "ordenar" o "comparación", y se acumula puntuación a lo largo de las pruebas para determinar un ganador final). Habrá más modos de juego en el futuro. De momento, solo esos dos.
- Pruebas tipo "test": las actuales, con temporizador, de reconocer los países en el mapa, de capitales, de saber las N ciudades más pobladas por continente, o del mundo... Vamos a generar algunos de estos tests. También banderas, etc. Se guardan puntuaciones para el usuario. Se generan rankings con puntuaciones de otros usuarios para cada test.

18. CONSTRUCCIÓN DE LOS DATASETS
- CSV maestro por entidad.
- Incluir continente, país, capital, población, superficie, idioma, moneda, indicadores clave, etc.
- Estadísticas necesarias: PIB, PIB per cápita, población, superficie, densidad, esperanza de vida, índice de desarrollo humano, índice gini, turismo, temperatura media, máxima, mínima... datasets extensos y completos con >50 variables. Filtrado a 12-14 variables "clave" para las pruebas por defecto.
